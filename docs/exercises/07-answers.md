# 阶段 7 参考答案：分布式锁、Lua 与 Stream

## 概念题参考答案

1. 防止服务宕机或线程异常后锁永远不释放，造成死锁。
2. 如果锁过期后被别人获取，旧线程再删除 key 就会误删别人的锁，所以必须校验 value。
3. Redisson 支持可重入、自动续期、Lua 原子释放、看门狗机制，使用上也更可靠。
4. 秒杀资格判断包含查库存、查是否下单、扣 Redis 库存、记录用户、写消息，这些操作必须原子执行，Lua 能保证脚本执行期间不被插入。
5. ACK 表示消息处理完成；Pending List 保存已投递但未确认的消息，方便异常后重新处理。

## 代码阅读题参考答案

1. `0` 表示成功获得资格；`1` 表示库存不足；`2` 表示重复下单。
2. 只有数据库扣库存和创建订单都成功后才 ACK。处理失败不能 ACK，否则消息丢失。

## 小改造参考

接口线程：

1. 生成订单 id。
2. 执行 Lua。
3. Lua 判断库存和一人一单。
4. Lua 扣 Redis 库存、记录用户、写入 Stream。
5. 接口立即返回订单 id。

后台线程：

1. 从 Stream 消费消息。
2. 转成 `VoucherOrder`。
3. 加用户分布式锁。
4. 数据库扣库存并创建订单。
5. 成功 ACK。
6. 异常处理 Pending List。

## 面试表达参考

“秒杀接口不直接写数据库，而是先生成订单 id，然后执行 Lua 脚本。Lua 在 Redis 中原子地判断库存、判断用户是否下过单、扣减 Redis 库存、记录用户并写入 Stream。接口拿到成功结果后立即返回订单 id。后台有独立线程从 Stream 消费订单消息，使用分布式锁和数据库事务完成扣库存和创建订单，成功后 ACK。如果处理失败，消息进入 Pending List，后续会重新拉取处理。”
